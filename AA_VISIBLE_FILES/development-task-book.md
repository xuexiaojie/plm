- 项目名称：工业炉计算平台

- 版本：v4.0

## 2. 编写原则

1. 本任务书仅定义本轮 AI 开发范围，不承接任何历史平台实现状态、既有完成度描述和兼容承诺。
2. 本任务书中的 V1.0 范围、对象关系、接口契约、状态机、样例数据和验收用例，均作为 AI 生成代码的硬约束。
3. 未在 V1.0 范围矩阵中标记为“实现”的功能，一律只保留规划说明，不进入本轮代码生成。
4. AI 开发按“先数据和协议，后服务和页面，再测试和文档”的顺序推进。

## 3. 建设目标

### 3.1 总体目标

建设一套面向工业炉专业计算的统一 Web 平台，覆盖项目建模、模板化计算、执行记录、审批、报告和横向对比，形成首批三类工业炉模型的可运行闭环。

### 3.2 本轮目标

1. 完成 V1.0 最小可运行闭环。
2. 固化 AI 开发所需的范围矩阵、核心对象、模板 Schema、执行器协议、审批状态机、报告规则和验收用例。
3. 支持梁式步进炉、辊底炉、环形炉三类首批模型的统一接入。
4. 让 AI 能基于明确边界生成数据库、接口、页面、报告和测试代码。

## 6. 名目定义

1. 名目是项目内的业务计算单元，用于承载一套独立的工业炉计算树。
2. 一个项目可包含多个名目，一个名目只属于一个项目。
3. 一个名目对应一个炉型实例或一个明确的计算对象，如“1780 加热炉改造方案 A”或“1 号辊底炉热处理制度校核”。
4. 名目下有且仅有一棵根结构树，树内节点按父子关系组织。
5. 名目字段至少包括：`code`、`name`、`furnace_type`、`business_scope`、`status`、`description`。

## 7. 核心对象关系

### 7.1 关系定义

1. `Project`：项目主实体。
2. `ProjectItem`：项目内名目实体。
3. `CalcNode`：名目结构树节点，节点类型分为目录节点和计算节点。
4. `CalcStepTemplate`：可复用计算步骤模板，定义算法入口、输入输出 Schema、报告模板和适用范围。
5. `CalcExecution`：一次计算执行记录。
6. `CalcResult`：一次执行的标准化结果。
7. `ApprovalRequest`：对结果或报告发起的审批申请。
8. `ApprovalLog`：审批流转日志。
9. `GeneratedReport`：草稿报告、正式报告、归档报告的元数据。
10. `ComparisonGroup`：横向对比分组。
11. `ProjectArtifact`：项目资料，类型包括现场反馈、审图单、技术附件、图纸目录。
12. `AiAnalysis`：AI 联合分析记录，保存请求上下文、大模型返回和 Provider 状态。

### 7.2 关系约束

1. `Project` 1:N `ProjectItem`
2. `ProjectItem` 1:1 根 `CalcNode`
3. `CalcNode` 1:N 子 `CalcNode`
4. 计算节点 N:1 `CalcStepTemplate`
5. `CalcNode` 1:N `CalcExecution`
6. `CalcExecution` 1:1 `CalcResult`
7. `CalcExecution` 1:N `ApprovalRequest`
8. `ApprovalRequest` 1:N `ApprovalLog`
9. `CalcExecution` 1:N `GeneratedReport`
10. `Project` 1:N `ProjectArtifact`
11. `ProjectItem` 1:N `ProjectArtifact`，`project_item_id` 可为空，表示项目级资料。
12. `Project` 1:N `AiAnalysis`

## 7.4 项目资料与 AI 联合分析

V1.0 必须支持项目级资料录入，资料类型固定为：

1. `site_feedback`：现场反馈。
2. `drawing_review`：审图单。
3. `technical_attachment`：技术附件。
4. `drawing_catalog`：图纸目录。

项目资料必须支持单条录入和批量录入。批量录入接口按项目提交，单条资料的 `project_item_id` 可为空；为空时表示资料先挂项目，后续可用于同一项目下设备或名目的 AI 联合分析。

AI 联合分析必须支持同一设备维度，例如“装出钢机”，将计算执行结果、现场反馈、审图单、技术附件、图纸目录组合为一个分析上下文。分析结论必须按物质流、能量流、信息流三个维度组织，再给出风险点、矛盾点和建议。正式调用方式为外部大模型 API，环境变量包括 `AI_API_URL`、`AI_API_KEY`、`AI_MODEL`。未配置 API 时允许返回本地演示分析，用于开发和验收环境。

## 10. 执行器统一协议

### 10.1 请求协议

```json

  "execution_id": "EXEC-20260603-0001",
  "template_code": "walking_beam_temp_profile_v1",
  "template_version": "1.0.0",
  "mode": "simulate",
  "inputs": {},
  "files": []
}
```

### 10.2 响应协议

```json
{
  "success": true,
  "feasible": true,
  "outputs": {},
  "warnings": [],
  "errors": [],
  "charts": [],
  "logs": []
}
```

### 10.3 协议约束

1. 所有执行器都必须使用统一 JSON 协议。
2. 执行器必须输出结构化 `warnings` 和 `errors`。
3. 执行器失败时也必须返回可解析的错误结构。
4. V1.0 只实现 Python 执行器，C# 字段和扩展点只保留协议位。

## 11. 审批与报告规则

### 11.1 审批状态机

审批状态固定为：

1. `DRAFT`
2. `SUBMITTED`
3. `RETURNED`
4. `APPROVED`
5. `ARCHIVED`
6. `CANCELLED`

### 11.2 流转规则

1. `DRAFT -> SUBMITTED`
2. `SUBMITTED -> RETURNED`
3. `SUBMITTED -> APPROVED`
4. `RETURNED -> SUBMITTED`
5. `APPROVED -> ARCHIVED`
6. `DRAFT -> CANCELLED`
7. `RETURNED -> CANCELLED`

### 11.3 报告状态

1. 草稿报告：执行完成后可生成，带“草稿”水印。
2. 待审报告：提交审批后生成，带“待审”水印。
3. 正式报告：审批通过后生成，获得正式编号。
4. 归档报告：归档后锁定下载内容。

### 11.4 报告规则

1. 报告编号在正式报告生成时分配。
2. 每次执行必须保留报告版本号。
3. 正式报告的输入、输出和审批记录必须可追溯。
4. 未审批通过的报告只能以草稿或待审状态下载。

### 15.5 统一错误码

V1.0 必须使用统一错误结构和错误码：

1. `PARAM_INVALID`：参数不合法。
2. `TEMPLATE_NOT_FOUND`：计算模板不存在。
3. `EXECUTION_FAILED`：执行器运行失败。
4. `REPORT_FAILED`：报告生成失败。
5. `PERMISSION_DENIED`：权限不足。
6. `NOT_FOUND`：资源不存在。
7. `STATE_INVALID`：状态流转不合法。
8. `FILE_INVALID`：文件类型或内容不合法。
9. `AI_PROVIDER_UNAVAILABLE`：AI Provider 不可用。

错误响应必须包含：`code`、`message`、`details`、`request_id`。

### 15.8 命名规范

V1.0 命名统一使用以下规则：

1. 数据库表名使用 `snake_case`。
2. Python 变量、字段和 JSON 字段使用 `snake_case`。
3. API 路径使用复数资源名，如 `/api/projects`。
4. 炉型编码固定为 `walking_beam_furnace`、`roller_hearth_furnace`、`ring_furnace`。
5. 模板编码格式为 `{furnace_type}_{calc_subject}_v{major}`。
6. 执行编号格式为 `EXEC-{yyyyMMdd}-{sequence}`。
7. 报告编号格式为 `RPT-{yyyyMMdd}-{sequence}`。

### 15.9 轻量 UI 规范

V1.0 页面采用轻量 HTML/JS 风格，必须统一：

1. 顶部导航显示门户、项目、名目、结构树、计算、审批、报告、对比、AI 分析。
2. 表单采用单列或双列布局，必填字段显示明确标记。
3. 主表格必须提供空状态、加载状态和错误状态。
4. 主操作按钮放在页面右上或表单底部。
5. 危险操作必须二次确认。
6. AI Mock 结果必须显示“演示分析”标识。
7. 报告草稿、待审、正式、归档状态必须有可区分视觉标签。

### 15.12 必须解决方案清单

本节根据审查截图和公开资料检索结果补齐。联网检索参考方向包括：工业炉与加热炉计算软件通常强调炉型结构、燃烧、传热、热平衡、物性、单位体系和报告输出；AI 开发规范通常强调规格先行、API 契约、输入输出约束、验收用例、追溯矩阵和分层验证；计算软件验证通常强调需求、设计、测试、报告、审计追踪和数据完整性。

#### 15.12.1 方案一：业务概念定型方案

必须先把业务概念写成可编码定义，再进入代码生成。

1. `Project` 是项目容器，承载合同、工程、设计或运行复核任务。
2. `ProjectItem` 是名目，代表项目内一个炉型实例或一个独立计算对象。
3. `CalcTree` 是名目下的计算编排结构。
4. `CalcNode` 是计算树节点，目录节点只组织结构，计算节点绑定模板。
5. `CalcStepTemplate` 是可复用计算模板，承载算法、输入、输出、报告和审批规则。
6. `CalcExecution` 是一次执行任务，记录输入、输出、状态、日志和快照。
7. `CalcResult` 是标准化计算结果，支持报告、审批和横向对比。
8. 页面展示必须按“项目列表 -> 名目列表 -> 计算树 -> 节点执行 -> 结果报告”展开。

#### 15.12.2 方案二：名目落地方案

名目必须按业务计算对象建模，字段和页面统一。

1. 字段：`code`、`name`、`project_id`、`furnace_type`、`business_scope`、`design_stage`、`status`、`description`。
2. 约束：同一项目内 `code` 唯一。
3. 关系：一个名目只属于一个项目，一个名目只对应一棵计算树。
4. 示例：项目“1780 热轧产线改造”下可有名目“1 号步进炉设计校核”“2 号辊底炉热处理复核”。
5. 页面：名目页必须展示所属项目、炉型、业务范围、计算树入口、最近执行状态。

#### 15.12.3 方案三：规格先行方案

所有开发任务执行前必须先形成结构化规格。

1. 先写 `00-version-scope.md`，锁定版本边界。
2. 再写 `01-domain-model.md`，锁定业务对象关系。
3. 再写 `02-database-schema.md` 和 `06-api-contract.md`，锁定数据和接口。
4. 再写 `09-acceptance-cases.md`，锁定验收用例。
5. 代码生成只能基于已经冻结的规格文件执行。
6. 规格变更必须先改规格文件，再改代码和测试。

#### 15.12.4 方案四：工业炉模型边界方案

首批三类模型必须明确专业计算边界。

1. 梁式步进炉：重点覆盖步距、步进周期、分区炉温、钢坯升温、表里温差、出炉温度。
2. 辊底炉：重点覆盖辊速、分区炉温、工件停留时间、热处理目标温度、升温均匀性。
3. 环形炉：重点覆盖旋转周期、环形分区、装出料节奏、炉温制度、温度均匀性。
4. 每类模型必须注明公式来源、简化假设、适用材料、适用尺寸、适用温度区间。
5. 专业公式缺来源时，该公式只能作为占位或 Mock，页面和报告必须标识“待专业校核”。

#### 15.12.5 方案五：单位与物性治理方案

工业炉计算必须统一单位、精度和物性来源。

1. 所有输入字段必须声明单位、范围、精度和默认值。
2. 所有输出字段必须声明单位、精度、正常范围和报告显示规则。
3. 单位换算必须集中实现，并提供测试用例。
4. 材料物性至少包含密度、比热、导热系数、发热值或相关热工参数。
5. 物性数据必须记录来源、版本和适用范围。

#### 15.12.6 方案六：执行与追溯方案

每次计算必须形成可复现记录。

1. 执行前保存输入快照、模板版本和执行器版本。
2. 执行后保存输出快照、日志、警告、错误和耗时。
3. 结果进入审批、报告和对比时必须引用同一 `execution_id`。
4. 模板变更后历史执行结果继续关联旧模板版本。
5. 对不可行工况保存真实输出、超限值和建议调整方向。

#### 15.12.7 方案七：API 契约方案

所有接口必须达到可直接生成代码和测试的精度。

1. 每个端点必须包含方法、路径、请求体、响应体、成功码、错误码、权限和示例。
2. 请求字段必须包含类型、必填、枚举、范围、默认值和说明。
3. 响应字段必须与数据库字段、页面字段、测试断言保持同名。
4. 错误码必须使用第 15.5 节统一错误码。
5. API 契约必须进入验收用例追溯矩阵。

#### 15.12.8 方案八：报告与审批方案

报告必须满足可审、可追、可锁定。

1. 草稿报告可由执行结果生成，带草稿水印。
2. 待审报告由提交审批动作生成，带待审水印。
3. 正式报告由审批通过动作生成，分配正式编号。
4. 归档报告由归档动作生成，锁定文件内容和下载记录。
5. 报告必须包含项目、名目、模板、输入摘要、输出摘要、约束判定、审批记录、生成时间和版本。

#### 15.12.9 方案九：验证与追溯矩阵方案

必须建立需求、设计、接口、测试、验收之间的映射。

1. 每条 V1.0 需求必须有唯一编号。
2. 每条需求必须映射至少一个设计条目。
3. 每个 API 必须映射至少一个测试用例。
4. 每个验收用例必须映射到需求编号和接口编号。
5. 验收文档必须区分输入验证、实现验证、工程验证、交互验证、回归验证和人工验收。

#### 15.12.10 方案十：AI 开发防跑偏方案

AI 生成必须按小颗粒任务推进。

1. 每次只允许生成一个层级的产物，如数据库、接口、页面、测试中的一种。
2. 每次生成前必须引用对应规格文件。
3. 每次生成后必须运行对应验证命令或人工核对清单。
4. 页面和接口字段发生不一致时，以 API 契约和数据库 Schema 为准，并回写页面需求卡片。
5. 业务定义发生不一致时，以第 6 章名目定义和第 7 章核心对象关系为准。

#### 15.12.11 方案十一：AI 分析边界方案

V1.0 AI 分析按演示分析处理，必须避免误导用户。

1. 页面显示“演示分析”。
2. 报告显示“AI 结论仅用于辅助判断”。
3. Mock 输出必须基于输入、输出和约束判定生成。
4. AI 分析不得替代专业审查和审批结论。
5. 真实 Provider 接入列入 V2.0。

#### 15.12.12 方案十二：可运行演示方案

系统必须内置一套可直接验收的演示数据。

1. 一个样例项目。
2. 三个样例名目。
3. 三棵样例计算树。
4. 三个首批模型模板。
5. 每个模型三组输入：正常、边界、不可行。
6. 每个模型对应预期输出范围。
7. 一条审批通过路径。
8. 一份正式报告样例。
9. 一个横向对比样例。
10. 一个 AI 演示分析样例。
