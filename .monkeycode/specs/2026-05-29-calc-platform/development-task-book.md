## 2. 项目背景

本项目用于建设一套面向工艺与过程计算的项目级计算平台。平台需要围绕“项目下名目”组织计算结构，支持树形编排、参数录入、双语言执行、审批、横向对比和 AI 智能分析，并与现有项目、用户、现场反馈和共享文件上下文联动。

当前系统已经形成可运行后端和轻量页面，前端入口拆分为录入台、计算台和分析台，具备继续迭代的基础。

## 3. 建设目标

### 3.1 总体目标

完成一套可运行、可演示、可继续扩展的计算平台基础版本，形成从建模、执行、审批、分析到 AI 诊断的业务闭环。

### 3.2 本阶段目标

1. 完成项目、名目、树节点、参数、步骤、执行、审批、对比、AI 的核心链路。
2. 提供轻量 Web 页面，满足录入、计算、查询分析三类场景。
3. 接入现场反馈上下文，恢复可见入口与项目级查询能力。
4. 保持 SQLite 可直接运行，同时兼容 PostgreSQL 演进。

## 4. 业务范围

### 4.1 范围内

1. 项目管理
2. 名目管理
3. 结构树管理
4. 全局参数和项目参数管理
5. Python 与 C# 执行器接入
6. 单节点执行与整树执行
7. 执行结果存储与查询
8. 多级审批流
9. 对比组与报表接口
10. AI 分析请求与结果查询
11. 现场反馈录入与项目级展示
12. 录入台、计算台、分析台三类页面

### 4.2 暂不纳入

1. 独立前端工程
2. 复杂分布式任务调度
3. 第三方 BPM 引擎
4. 文件中心完整实现
5. 现场反馈附件与富文本能力
6. 名目级和节点级现场反馈细粒度绑定

## 5. 当前实现现状

### 5.1 已完成能力

1. FastAPI + SQLAlchemy 基础后端骨架已完成。
2. `Project`、`ProjectItem`、`CalcNode`、`CalcStep`、`CalcExecution`、`CalcResult`、`ApprovalRequest`、`ApprovalLog`、`ComparisonGroup`、`ComparisonItem`、`AiAnalysisRequest`、`AiAnalysisResult` 已完成。
3. 全局参数、项目参数、步骤、输入引用、树节点、执行结果等接口已完成。
4. Python 执行器已可运行脚本，C# 执行器已提供基础接入形式。
5. 整树执行、结果查询、审批通过拒绝、对比组报表、AI 分析请求与结果查询已完成。
6. 门户页 `/`、录入台 `/entry`、计算台 `/compute`、分析台 `/analysis` 已完成。
7. AI 分析已接入三流理论结构化输出。
8. 现场反馈已补齐项目级最小闭环：模型、接口、录入入口、分析展示、AI 上下文摘要。
9. 测试当前通过，`pytest` 为 `4 passed`。

### 5.2 当前不足

1. 结构树仍缺少复制、逻辑删除恢复、版本管理等增强能力。
2. 录入台仍缺少项目参数、名目、树节点的直接表单化录入。
3. 计算台仍缺少单节点执行、执行历史列表、轮询刷新等增强能力。
4. 分析台仍缺少对比面板、筛选、时间线等能力。
5. 现场反馈仍是项目级最小模型，缺少更细粒度关联和编辑管理能力。
6. AI provider 仍为 stub，尚未接入真实外部模型服务。
7. Alembic 迁移与运行时补表机制仍需统一治理。

## 6. 目标系统结构

### 6.1 页面结构

1. 门户页：统一导航和系统定位说明。
2. 录入台：创建项目、全局参数、现场反馈，后续扩展项目参数、名目、树节点录入。
3. 计算台：按项目和名目执行结构树，查看最近结果。
4. 分析台：查看结构树、项目参数、现场反馈、审批和 AI 分析结果。

### 6.2 后端结构

1. `app/main.py` 承载 API 和轻量页面。
2. `app/models.py` 管理核心 ORM 模型。
3. `app/schemas.py` 管理请求与响应结构。
4. `app/executors.py` 管理执行器。
5. `app/ai_service.py` 管理 AI provider 抽象与 stub 实现。

## 7. 开发任务拆解

## 7.1 阶段一：基础能力补齐

### 目标

让录入、建模、执行、分析形成更顺畅的基础闭环。

### 任务

1. 在录入台补充项目参数录入表单。
2. 在录入台补充名目录入表单。
3. 在录入台补充树节目录入表单。
4. 补充项目详情展示，包括负责人、状态、现场反馈范围。
5. 补充项目级现场反馈列表刷新与错误提示。

### 交付物

1. 页面可直接录入项目、参数、名目、节点、现场反馈。
2. 对应 API 使用说明或 README 更新。

## 7.2 阶段二：计算能力增强

### 目标

让计算台真正成为主要执行入口。

### 任务

1. 增加单节点执行入口。
2. 增加最近执行历史展示。
3. 增加结果刷新与简单轮询。
4. 展示执行耗时、状态、日志摘要。
5. 增加执行失败提示与错误码展示。

### 交付物

1. 计算台支持单节点和整树执行。
2. 结果展示更完整。

## 7.3 阶段三：审批流完善

### 目标

让审批成为可视、可查、可控的业务链路。

### 任务

1. 增加撤回接口与页面操作入口。
2. 增加审批日志查询展示。
3. 增加审批状态与执行权限联动规则。
4. 在分析台展示多级审批进度。

### 交付物

1. 完整审批动作集合。
2. 审批日志可查。

## 7.4 阶段四：现场反馈深化

### 目标

让现场反馈从“项目级上下文”演进为“业务级分析输入”。

### 任务

1. 为现场反馈增加时间、标签、来源扩展字段。
2. 支持现场反馈关联到名目。
3. 支持现场反馈关联到树节点。
4. 在分析台增加现场反馈筛选与排序。
5. 在 AI 上下文中加入反馈摘要裁剪策略。

### 交付物

1. 现场反馈支持更细粒度分析。
2. AI 分析使用更准确的现场上下文。

## 7.5 阶段五：横向对比增强

### 目标

把现有对比接口升级为可展示、可解释的分析模块。

### 任务

1. 在分析台增加对比组列表和报表面板。
2. 增加结果指标提取展示。
3. 增加按 `step_type` 的项目聚合筛选。
4. 增加导出接口或 JSON 下载入口。

### 交付物

1. 分析台可直接查看横向对比结果。

## 7.6 阶段六：AI 能力增强

### 目标

把现有 stub 分析升级为真实可扩展能力。

### 任务

1. 抽象 provider 配置方式。
2. 接入真实 AI 服务。
3. 为 AI 请求增加上下文摘要、超时、失败状态控制。
4. 在分析台展示更多结构化分析字段。
5. 保留三流理论结构输出兼容。

### 交付物

1. AI 分析可切换真实 provider。
2. 页面可展示更完整诊断信息。

## 7.7 阶段七：工程治理

### 目标

提高系统可维护性和稳定性。

### 任务

1. 将 `@app.on_event("startup")` 迁移为 lifespan。
2. 补齐 Alembic 迁移，减少运行时补字段和补表逻辑。
3. 增加更多 API 测试覆盖。
4. 梳理 README 的运行、迁移、测试说明。
5. 为页面拆分后的入口补充更清晰导航文案。

### 交付物

1. 迁移和运行方式更稳定。
2. 测试覆盖更完整。

2. `python3 -m compileall app tests` 通过。
3. SQLite 环境可直接运行。
4. Alembic 迁移可执行。

## 10. 风险与应对

### 风险一：SQLite 与迁移状态不一致

- 应对：保持迁移脚本幂等，逐步减少运行时补字段逻辑。

### 风险二：页面拆分后入口认知成本上升

- 应对：加强门户说明、页面互链和空状态提示。

### 风险三：AI 上下文持续膨胀

- 应对：对结果、文件、现场反馈采用摘要和数量限制。

### 风险四：执行器行为不统一

- 应对：统一 stdout JSON 协议和错误码。
