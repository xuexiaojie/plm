# 计算平台需求文档

## 1. 文档信息

- Feature: 计算平台
- Date: 2026-05-29
- Source: 用户提供的原始需求整理

## 2. 目标

系统需要提供一套面向工艺与过程计算的项目级计算平台，支持树形编排、双语言执行、统一参数与结果管理、审批流、跨项目对比分析，以及 AI 智能分析能力。

## 3. 术语

- 项目：承载一次完整计算业务的数据域，关联结构树、参数、结果、文件、AI 分析记录。
- 名目：项目下的业务分类单元，计算树按项目下名目组织。
- 结构树：由根节点、分支节点和叶子节点组成的可编排计算结构。
- 计算步骤：可被节点绑定的独立计算模块定义。
- 计算执行：按树结构和依赖关系调度节点对应的计算步骤。
- 审批：对节点定义、计算结果或发布动作进行状态流转和人工审查。
- 对比组：用于跨项目横向比较同类计算步骤结果的逻辑集合。

## 4. 业务范围

### 4.1 In Scope

- 项目与名目管理
- 结构树创建、编辑、排序、移动、复制
- 叶子节点与计算步骤绑定
- Python 与 C# 双语言执行
- 参数录入、导入、引用、类型转换
- 结果持久化、查询、审批、发布
- 多级审批流
- 同类计算跨项目对比
- 基于项目参数、结果、文件和现场反馈共享数据的 AI 分析

### 4.2 Out Of Scope

- 第三方 BPM 引擎接入
- 在线 C# IDE 式编辑体验
- 复杂分布式任务编排集群
- AI 模型训练与模型管理

## 5. 角色

- 平台管理员：维护全局参数、步骤模板、审批规则。
- 项目负责人：创建项目与名目，组织结构树，发起执行与提审。
- 计算工程师：编写计算步骤，配置输入引用，查看运行结果。
- 审批人：审核节点、结果或发布申请并给出意见。
- 分析人员：查看横向对比报告和 AI 分析结果。

## 6. 功能需求

### 6.1 项目与名目管理

1. 系统应支持创建、查看、更新项目。
2. 系统应支持在项目下创建名目，并以名目作为计算树归属单元。
3. 系统应支持项目与用户名复用现有现场反馈模块中的主体信息。
4. 系统应支持项目级共享文件与共享数据供计算和 AI 分析使用。

### 6.2 结构树管理

1. 系统应支持为项目名目创建一棵结构树。
2. 系统应支持新增根节点、子节点和平级节点。
3. 系统应支持节点改名、排序、拖拽移动、复制和逻辑删除。
4. 系统应支持节点类型区分目录节点与计算节点。
5. 系统应支持任意层级扩展。
6. 系统应支持保存结构树草稿与恢复历史版本。

### 6.3 计算步骤管理

1. 系统应支持创建和维护计算步骤。
2. 系统应支持步骤语言选择，至少包括 Python 与 C#。
3. 系统应支持步骤定义入口点、脚本内容、可执行文件路径或程序集路径。
4. 系统应支持步骤分类 `step_type`，用于跨项目横向对比。
5. 系统应支持配置输入引用、默认参数和输出 Schema。

### 6.4 参数管理

1. 系统应支持全局参数与项目参数两级管理。
2. 系统应支持参数类型 `number`、`text`、`bool`。
3. 系统应支持 Excel 批量导入参数。
4. 系统应支持节点执行时从全局参数、项目参数、上游结果和共享文件元数据中解析输入。

### 6.5 执行调度

1. 系统应支持单节点执行。
2. 系统应支持整棵树按结构顺序执行。
3. 系统应支持在执行前构建统一上下文。
4. 系统应支持按步骤语言选择对应执行器。
5. 系统应支持保存执行日志、执行耗时、输入快照和结果快照。
6. 系统应支持审批状态参与执行权限控制。

### 6.6 审批流

1. 系统应支持对节点配置和结果发起提审。
2. 系统应支持审批通过、拒绝、撤回。
3. 系统应支持多级审批状态流转。
4. 系统应支持记录审批意见与审批日志。
5. 系统应支持根据审批结果控制结果可见性与可发布性。

### 6.7 横向对比

1. 系统应支持按 `step_type` 选择同类计算步骤。
2. 系统应支持跨项目聚合同类结果。
3. 系统应支持指标表格对比、时间趋势对比和结果导出。
4. 系统应支持保存对比组与对比项。

### 6.8 AI 智能分析

1. 系统应支持发起 AI 分析请求。
2. 系统应支持 AI 分析使用项目参数、计算结果、结构树、共享文件和现场反馈共享数据。
3. 系统应支持返回诊断、建议、风险提示和可追溯分析记录。
4. 系统应支持查看 AI 分析历史。

## 7. 非功能需求

### 7.1 安全

1. 计算执行环境应与主服务隔离。
2. 步骤执行输入与输出应进行 Schema 校验。
3. 审批、执行、AI 调用都应保留审计日志。

### 7.2 性能

1. 单项目结构树在 1000 个节点规模内应可流畅展示与编辑。
2. 整树执行应支持异步后台任务。
3. 对比查询应支持分页与筛选。

### 7.3 可扩展性

1. 执行器应支持后续扩展更多语言。
2. 审批流应支持规则配置。
3. AI 接口应支持后续替换底层模型服务。

### 7.4 可维护性

1. 结构树、步骤、审批、执行、AI 分析模块应边界清晰。
2. 数据模型应兼容 PostgreSQL，开发环境可运行在 SQLite。

## 8. 核心业务流程

### 8.1 建模流程

1. 创建项目与名目。
2. 搭建结构树。
3. 创建或绑定计算步骤。
4. 配置输入引用和参数。

### 8.2 执行流程

1. 用户发起单节点或整树执行。
2. 系统校验审批状态和输入完整性。
3. 调度器按顺序构建执行计划。
4. 执行器返回标准化结果。
5. 系统保存结果、日志与快照。

### 8.3 审批流程

1. 用户对节点或结果发起提审。
2. 系统分配审批节点。
3. 审批人给出通过或拒绝意见。
4. 系统更新状态并影响后续执行与发布权限。

### 8.4 对比分析流程

1. 用户选择 `step_type` 与项目集。
2. 系统聚合同类结果。
3. 系统生成报表与导出内容。

### 8.5 AI 分析流程

1. 用户选择分析对象与上下文范围。
2. 系统组装结构、参数、结果、文件和现场反馈数据。
3. AI 服务返回诊断与建议。
4. 系统持久化结果并支持追溯。

## 9. 验收标准

1. 用户可以在项目名目下创建并保存多层树结构。
2. 叶子节点可以绑定 Python 或 C# 计算步骤并执行。
3. 执行结果可以统一存储、查询和审批。
4. 审批状态可以控制结果发布和部分执行权限。
5. 用户可以按同类步骤跨项目生成对比结果。
6. AI 分析可以基于共享上下文输出诊断建议。
